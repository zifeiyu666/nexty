# One Custom Song SEO 调查报告

审计日期：2026-07-12  
审计对象：`https://onecustomsong.com` 当前项目源码  
范围：可索引静态页面、英文博客、本地可见的多语言页面、站点地图、robots、metadata、正文主题与源码内链。

## 1. 结论摘要

综合评分：**71/100（基础良好，但存在页面定位与索引治理缺口）**。

| 维度 | 得分 | 判断 |
|---|---:|---|
| 抓取与索引 | 72/100 | robots 与 sitemap 已配置，但 sitemap 漏页、错误/账户页索引规则不完整 |
| 页面关键词定位 | 74/100 | 商业页主题总体明确，部分页面彼此抢同一搜索意图 |
| Title / Description / H1 | 77/100 | 主要落地页较好；主页 Title 偏长，部分工具页 Title 太泛 |
| 内容与搜索意图 | 73/100 | 商业、信息、工具三类内容齐全；个别“工具页”与内容落地页复用导致意图不纯 |
| 内链与主题集群 | 65/100 | Playlist 集群优秀；核心商业页之间缺少系统化上下文互链，部分 sitemap 页面弱链接 |
| 国际化 SEO | 70/100 | canonical/hreflang 框架合理；部分语言内容并未真正本地化，却默认声明所有语言版本 |

最重要的判断：

1. `meta keywords` 为空不是问题。Google 不使用该标签进行排名；Google 会从 Title、H1、首屏正文、H2、锚文本、URL、结构化数据和站点主题关系判断关键词。
2. 网站的核心主题清晰：**custom song / personalized song gift**。但主页同时承载品牌词、品类词、AI 定制歌曲和礼物意图，应该明确以 `custom song gift` 为主，`personalized song gift` 与 `AI custom song` 为辅。
3. 最大的关键词蚕食发生在生日主题：商业落地页 `/occasions/custom-happy-birthday-song` 与博客 `/blog/custom-happy-birthday-song` 均强打 `custom happy birthday song`。应让落地页承接交易意图，博客承接“ideas / examples / ages / recipients”等信息意图，并调整博客 Title/H1。
4. `/ai-song-lyric-generator` 的 metadata 声称是生成器，但实际复用了 `CustomSongLyricGiftsPage`。如果页面首屏和正文仍偏“lyric gifts”，Google 可能判断它不是纯工具页，影响 `AI song lyric generator` 的相关性与用户行为信号。
5. sitemap 未包含若干公开营销页，如 `/music-video-gift-maker`、`/lyricwallart`、`/samples`、`/about`；其中 samples 已 noindex，不应加入，但前三者应按索引策略明确处理。

## 2. 方法与证据边界

本报告模拟 Google 的页面主题理解，不声称模拟其私有算法。判断依据为：

- 页面 Title、description、H1/H2、正文高频实体、URL 语义、图片 alt、内部链接锚文本。
- `app/sitemap.ts`、`app/robots.ts`、`lib/metadata.ts`、路由页面、英文文案与博客 MDX。
- 项目中的动态数据库博客/Glossary 内容无法在不依赖生产数据库的情况下完整枚举，因此相关页面只审计模板与本地英文文章。
- 未接入 Google Search Console、GA4、Keyword Planner、Ahrefs/Semrush。因此**不提供虚构的搜索量、难度、真实排名、展示量、点击率、收录数或 Core Web Vitals**。这些必须用线上数据补证。

Google 更可能把本站识别为以下主题树：

## 7. 内链结构调查

### 7.1 当前结构

从源码可验证的结构：

- 全局 Header 至少稳定链接首页和 `/create-song`；其余链接由运行时 `headerLinks` 传入，需结合渲染页最终确认。
- Footer 链接 `/about`、隐私、条款、退款，并根据配置输出文章链接。
- 首页强链到 `/create-song`、`/pricing`、`/blog`、occasion 页面以及产品工具页。
- Playlist hub → 三个目录页 → 详情页；详情页 → 创建页，并通过 `relatedSlugs` 建立横向相关推荐。这是当前最完整的主题集群。
- 博客详情模板稳定回链 `/blog`，CTA 稳定指向 `/create-song`；文章正文存在部分绝对内部 URL。
- 工具页 `/lyricwallart` 与 `/music-video-gift-maker` 主要链到 `/create-song` 和 `/login`，向相关博客与商业落地页的上下文链接不足。

### 7.2 内链问题

1. **链接权益过度汇聚到 `/create-song`**：转化合理，但 Google 得到的是“创建页最重要”，其他主题权威页得到的上下文支持偏少。
2. **生日内容冲突缺少明确的主从关系**：博客应该用描述性锚文本链接到生日商业页，商业页可链接博客的 ideas 指南，但两者锚文本必须区分交易与信息意图。
3. **工具集群未闭环**：歌词生成器、歌词写作博客、墙画工具、墙画 ideas 博客、音乐视频工具应互相形成语义链，而不是全部只导向创建歌曲。
4. **核心 occasion 页横向关系弱**：生日、周年、music gift 可通过“按场合选择”区块互链，并接入 Playlist 对应页。
5. **About 弱链接/漏 sitemap**：Footer 有链接，但内容和 metadata 太弱，难以承担实体信任页作用。
6. **绝对内部链接不利于多语言继承**：MDX 中 `https://onecustomsong.com/...` 会把西语/日语用户送回英文 canonical。应优先使用 locale-aware 相对链接或 MDX 内链组件。

## 8. 技术 SEO 调查

### 8.1 正面项

- 使用统一 `constructMetadata` 输出 canonical、Open Graph、Twitter、robots 与 hreflang。
- 默认语言 URL 不带 `/en`，`x-default` 指向英文版，结构合理。
- robots.txt 声明 sitemap，并阻挡 API、dashboard、payment、短链接等非搜索区域。
- sitemap 去重博客 URL，并排除一个旧 slug。
- 关键页面使用语义 H1，博客详情模板会使用文章 metadata title 渲染 H1。

### 8.2 需要修复的问题

| 优先级 | 问题 | SEO 影响 | 修复 |
|---|---|---|---|
| P0 | robots 阻挡不能替代 `noindex` | 已知 URL 仍可能以“无摘要”形式出现在索引中 | 登录、支付、错误、账户、动态私有内容在页面 metadata/X-Robots 中明确 noindex |
| P1 | sitemap 漏公开营销页 | 发现速度与站点结构信号变弱 | 加入 `/about`、`/music-video-gift-maker`、`/lyricwallart`；按意图决定 glossary 等 |
| P1 | sitemap 法律页 priority 0.8 | 向爬虫表达的相对重要性不合理（priority 本身影响很弱） | 调至 0.2–0.3；核心商业页保持 0.8–1.0 |
| P1 | hreflang 默认声明所有 locale | 若某页只做了英文回退或未真正翻译，可能出现低质量/近重复语言页 | 每页传真实 `availableLocales`；未完成翻译的 locale 不输出 alternate |
| P1 | `keywords: []` | 无排名损失，但生成空标签没有价值 | 可移除；不要投入时间填 meta keywords |
| P1 | 工具页正文与 metadata 意图不一致 | Google 对页面主题置信度下降 | 为 generator 创建专属工具正文与 FAQ，不复用 gift landing component |
| P2 | 动态内容 sitemap `lastModified` 回退到 `new Date()` | 每次生成都像内容刚更新，浪费重抓取 | 无真实日期时使用 publishedAt 或稳定构建日期，不用当前时间 |
| P2 | 绝对英文内链 | 多语言页面的链接关系被拉回英文 | 使用 locale-aware Link/相对路径 |
| P2 | sitemap 运行时依赖数据库 | 数据库故障可能导致 sitemap 500 | 对 DB 查询做容错并保留静态/本地条目 |
| P2 | 未见全站结构化数据策略 | 错失 Organization、WebSite、Breadcrumb、Article、FAQ 等明确实体信号 | 按页面类型输出有效 JSON-LD，并用 Rich Results Test 验证 |

### 8.3 Core Web Vitals 边界

源码包含大量图片、视频、动态编辑器与客户端交互，首页视频墙及工具编辑器是潜在 LCP/INP 风险。但没有真实 Lighthouse/CrUX 数据，不能给出性能结论。上线后应分别测试移动端：`/`、`/create-song`、生日页、一个博客、一个 Playlist 详情；目标为 LCP ≤ 2.5s、INP ≤ 200ms、CLS ≤ 0.1（75 分位）。

## 9. 内容质量与 E-E-A-T

优势：

- 本地博客多数有清晰作者视角、问题定义、示例、表格和具体流程。
- Founder story 可以支撑品牌真实性。
- 产品页面包含真实功能、价格方向、免费预览与交付物，而非空泛关键词页。

缺口：

- `best personalized song company` 属于商业比较内容，应披露评估方法、更新时间、比较对象数据来源，以及本站作为参评方的利益关系。
- About 页目前内容/metadata 太泛，应加入团队/创始人、产品为何存在、审核流程、联系方式、内容编辑原则和真实品牌信息。
- 博客文章 metadata 未看到明确作者、reviewedBy、Article schema 的统一实现；应加强作者页与更新政策。
- FAQ schema 只能用于页面真实可见 FAQ，且 Google 已大幅限制普通商业网站 FAQ 富结果，不应把它当作流量保证。

## 10. 分阶段实施计划

### 第 1 周：消除索引与关键词冲突

1. 给登录、支付成功、错误页、账户/用户资产页统一加 `noindex, follow`。

4. 将 `/ai-song-lyric-generator` 改成真正独立的歌词工具页内容。

### 第 2–3 周：重构关键词所有权与内链

1. 按本报告的关键词所有权重写主页、pricing、music gift、video、wall art 的 Title/H1。
2. 建立三个内容闭环：生日、周年、歌词/墙画工具。

### 第 4–6 周：实体、结构化数据与数据验证

2. 接入 Search Console，导出 page × query 数据验证本报告中的关键词推断。

## 12. 最终判断

当前关键词策略**方向合理，执行约七成成熟**。网站已经形成清晰品类与多个可扩展内容集群，尤其 Playlist 架构值得保留。当前不应该去“填满每页 meta keywords”，而应集中解决四件事：**关键词所有权、生日页蚕食、工具页搜索意图一致性、内链权重分配**。

最优先的三个动作是：

1. 让 `/occasions/custom-happy-birthday-song` 成为唯一 `custom happy birthday song` 交易权威页。
2. 为 `/ai-song-lyric-generator` 建立真正的生成器页面内容，不再复用礼物落地页语义。
3. 补齐 sitemap/noindex，并把博客、Playlist、工具页与对应商业页按主题闭环连接。

完成后，再用 Search Console 的真实 query/page 数据做第二轮校准，才能判断搜索量、排名提升和 Google 是否选择了预期页面。
